# Announcing Tidepool: Helius DAS, locally

**One binary, one npm package, zero API keys.**

Tidepool is a Helius-compatible local dev environment for Solana. Point your existing `helius-sdk` client at it instead of `mainnet.helius-rpc.com`, and your DAS calls, compressed-NFT proofs, Enhanced Transactions, WebSocket subscriptions, and webhook CRUD all resolve offline against a local validator. Same wire shapes, same SDK code path, no rate limits, no monthly bill.

## Why this exists

Two things broke "develop against Helius" as a tight dev loop:

- **Cost + rate limits.** Real Helius has a free tier, but it's coupled to internet, API keys, and a quota you'd rather not burn on test runs. CI pipelines can't sanely depend on it.
- **`confirmTransaction()` doesn't work on Surfpool.** Surfpool's native WebSocket doesn't implement `signatureSubscribe`, so `@solana/web3.js`'s confirmation polling hangs. Every `helius-sdk` method that composes "send, wait, assert" — `sendSmartTransaction`, `broadcastTransaction`, `pollTransactionConfirmation` — fails against the local validator alone.

Tidepool sits between your app and Surfpool. Methods Helius owns (DAS, cNFTs, Enhanced Transactions, webhooks, WS polyfills) get served from local state. Everything else falls through to the upstream unchanged.

## What's in v0.1.5

- **Full DAS surface.** `getAsset`, `getAssetBatch`, `getAssetProof`, the `getAssetsBy*` family, `searchAssets`, `getNftEditions`. MplCore + Token Metadata (Token and Token-2022) + Bubblegum cNFTs, all resolved from real on-chain bytes via the official Metaplex decoder crates.
- **Compressed-NFT indexer.** Local Bubblegum replay over `getSignaturesForAddress` + `getTransaction`. Authoritative leaf state from the noop-CPI `LeafSchemaEvent` — proofs match on-chain even after `setAndVerifyCollection` and friends.
- **`confirmTransaction()` that actually completes.** HTTP-polling polyfills for `signatureSubscribe`, `accountSubscribe`, `logsSubscribe` (mentions filter) on the WS port.
- **Enhanced Transactions** via REST (`/v0/transactions`, `/v0/addresses/{addr}/transactions`) with the parser shaped to match Helius's response exactly.
- **Webhook CRUD** simulator on `/v0/webhooks`.
- **Transport-faithful.** JSON-RPC, REST, and WebSocket all land at the same Helius URLs they would in production, so dev-only code can't compile that won't work against real Helius.

## Install

Pick your consumption story.

**As a binary:**
```bash
cargo install tidepool-cli
tidepool start --port 8897 --upstream http://127.0.0.1:8899
```

**In Node tests (MSW, Nock, undici, anything):**
```bash
npm install @vibestackmd/tidepool
```
```ts
import { HeliusContext, handleJsonRpcBody } from "@vibestackmd/tidepool";
```

**As a Rust library:**
```toml
[dependencies]
tidepool-rpc = "0.1"
```

Every release ships from GitHub Actions via OIDC trusted publishing on both registries, with signed sigstore provenance on the npm tarball. No long-lived API tokens involved at any step.

## Built on Surfpool

Tidepool works against any Solana RPC, but [Surfpool](https://github.com/txtx/surfpool) is recommended: its mainnet-forking means any real account you ask about "just works" without pre-declaring it. That's what makes the dev-loop feel magic instead of tedious. Mention to the Surfpool folks: signature subscribe was the one missing piece keeping the helius-sdk integration from working out of the box. Tidepool polyfills it.

## Status, honestly

This is a local development tool. Ship to real Helius in production. The library crates (`tidepool-core`, `tidepool-rpc`, `tidepool-server`) are internal surfaces until 1.0 — the supported entry point is `tidepool-cli` (and `@vibestackmd/tidepool` for JS).

Bugs, missing methods, weird DAS edge cases: [open an issue](https://github.com/vibestackmd/tidepool/issues). The contract-test rig caught 10+ Helius-vs-us drift bugs in the v0.6 → v0.1.x rewrite; new ones surface fast.
