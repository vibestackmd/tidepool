# Tidepool v0.4.0 — off-chain metadata, finally

**`getAsset` now returns the image.**

Until now, Tidepool's `getAsset` returned the on-chain half of an asset — `name` and the `uri` — but left `content.links.image`, `content.metadata.description`, `content.metadata.attributes`, and `content.files[]` empty. That's because an NFT's on-chain account only stores a name and a URL; everything you actually render (image, description, traits, extra files) lives in a JSON document at that URL.

Real Helius fetches that document and folds it into the response. That's the whole point of DAS — one call gives you the complete, merged asset. v0.4.0 brings Tidepool in line: `getAsset` now fetches the off-chain JSON and populates the full content block.

```jsonc
// before v0.4.0
"content": {
  "json_uri": "https://arweave.net/abc",
  "metadata": { "name": "Cool Asset" },
  "links": {},          // empty
  "files": []           // empty
}

// v0.4.0
"content": {
  "json_uri": "https://arweave.net/abc",
  "metadata": {
    "name": "Cool Asset",
    "description": "The description from the off-chain JSON",
    "attributes": [{ "trait_type": "Color", "value": "Blue" }]
  },
  "links": {
    "image": "https://arweave.net/img.png",
    "animation_url": "https://arweave.net/vid.mp4"
  },
  "files": [{ "uri": "https://arweave.net/img.png", "mime": "image/png" }],
  "category": "image"
}
```

## Built for CI from day one

A local dev tool that reaches out to arbitrary URLs is a flaky-test generator if you're not careful. v0.4.0's fetch is defensive:

- **Fail-soft.** A blocked egress (CI sandbox), a slow gateway, or a dead URI degrades the response to its on-chain fields. It never errors the `getAsset` call. A network-restricted CI run gets blank-but-valid responses, not failures.
- **`file://` support.** Locally-seeded metadata is fetched fully offline and deterministically — ideal for test fixtures.
- **`--no-offchain-metadata` flag** (env `TIDEPOOL_NO_OFFCHAIN_METADATA`) disables fetching entirely for hermetic runs.
- **2 MiB cap**, inherits the RPC client timeout, **cached per asset** (fetched once).

## Why this shipped now

This is the first concrete improvement from **dogfooding Tidepool into a real project**. Pointing a real Solana app at Tidepool surfaced the gap immediately: the app's UI rendered blank thumbnails because `content.links.image` was empty. That's not a project-specific quirk — *every* DAS consumer (a CLI listing NFTs, a mobile gallery, a marketplace, an explorer) needs the rendered image and description. Fetching off-chain metadata is core Helius-parity behavior, so it belongs in Tidepool for everyone, not in a per-project shim.

Two adjacent ideas were considered and deliberately left out, to avoid overfitting to one consumer:
- **Search-index-on-mint** (making `searchAssets` return freshly-minted assets without a prior `getAsset`) — stays a consumer-side concern. Tidepool can't continuously index a forked mainnet, and the lazy "fills as you fetch" model is a legitimate local-index design.
- **`searchAssets` `sortBy` / `showCollectionMetadata`** — real DAS features, backlog.
