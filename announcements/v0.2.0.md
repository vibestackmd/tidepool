# Tidepool v0.2.0 — Surfpool catch-up

**One thousand lines lighter. Same single endpoint. Real native subscriptions underneath.**

When Tidepool v0.1.x launched, Surfpool's WebSocket didn't implement Solana's subscription methods, so we wrote an HTTP-polling polyfill: every `signatureSubscribe` translated into a periodic `getSignatureStatuses` call, every `accountSubscribe` into `getAccountInfo`, every `logsSubscribe` (with `mentions` filter) into `getSignaturesForAddress` + `getTransaction` fan-out. It worked, and `confirmTransaction()` finally completed against a local validator. But the code was ~640 lines of polling state machines that we had to maintain against a moving target.

In Surfpool v1.1+, the upstream natively implements every subscription method we were polyfilling — plus `programSubscribe` and `slotSubscribe` we hadn't gotten to. The right move was clear: stop polyfilling, start proxying.

## What changed

**WS server is now a thin reverse proxy.** Per client, Tidepool opens one WS connection to Surfpool (default `ws://upstream:8900`) and pumps frames bidirectionally. Single-endpoint UX is preserved — clients still connect to `ws://tidepool:port+1` — but every subscription method Surfpool supports now works transparently, with full filter fidelity (including `logsSubscribe` with `all` / `allWithVotes`, not just the `mentions` shape our polyfill handled).

Three manifest entries (`signatureSubscribe`, `accountSubscribe`, `logsSubscribe`) move from compat level **SHIM → EXACT**.

**Net diff: ~1,150 lines deleted, ~180 added.** Roughly one polling state machine, three retry loops, a subscription registry, and 500 lines of polling integration tests, all gone.

## Compatibility note

The `compatibility.toml` pin tightens from `any` to `>=1.1, <2`. Users running Surfpool earlier than v1.1.0 must upgrade — the WS proxy depends on subscription methods that didn't exist before then.

## Also in this release

- Upstream references updated from `txtx/surfpool` to `solana-foundation/surfpool` (Surfpool moved orgs).
- CI fixes that fell out of v0.1.5's release iteration: `node --test` Node 24 incompatibility, `verify-changelog.sh` immediately-post-release false positive.
- README polish: live crates.io + npm version badges, fixed install snippet versions, Roadmap reflects what was actually shipped.
- Stale "WS polyfill" mentions scrubbed from active claims; historical mentions left as-is (they accurately describe what *was*).

## Install

```bash
cargo install tidepool-cli   # binary
npm install @vibestackmd/tidepool  # JS test integrations (MSW, Nock, undici)
```

Lockstep across both registries at 0.2.0. Provenance attestations on the npm tarball.

## What's next (0.3.x)

Helius has shipped a lot since Tidepool v0.1.x: `getTransactionsForAddress`, `getTransfersByAddress`, `getProgramAccountsV2`, `getTokenAccountsByOwnerV2`, the Wallet REST API, Sender JSON-RPC, Helius for Agents (MCP). Tidepool's "Helius-compatible" claim is starting to age. v0.3 is the catch-up release.
